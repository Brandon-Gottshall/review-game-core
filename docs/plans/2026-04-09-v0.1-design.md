# review-game-core v0.1 — Design

**Date:** 2026-04-09
**Status:** Approved, implementing
**Scope:** First extraction from `cs1301-review-game`. Lift the truly-generic primitives so other games (ASTR, future) can adopt them via git URL. Iterate per-game.

## Goals

1. Ship a source-only git-installable package with the shared primitives already used in `cs1301-review-game`.
2. Zero runtime dependencies, no framework lock-in at the core.
3. WF harness reusable across vitest and jest consumers.
4. v0.1 is intentionally small — each game onboarding may trigger breaking minor bumps. Consumers pin by tag.

## Non-goals (v0.1)

- No UI components. Consumers own their own React/Tailwind/whatever.
- No Java parser. CS-specific.
- No spaced repetition. Extract after ASTR validates the shape.
- No question generators for any particular domain. Each game writes its own.
- No pre-commit hooks or build pipeline. Source-only.

## Package shape

```
review-game-core/
├── src/
│   ├── index.ts              # public surface
│   ├── question/             # Question, InteractiveData (generic)
│   ├── concept/              # ConceptNode, mastery, unlock, populate
│   ├── generator/            # mulberry32, Generator<Q>
│   └── wf-harness/
│       ├── validators.ts     # pure, framework-agnostic
│       ├── vitest.ts         # vitest adapter
│       └── jest.ts           # jest adapter
├── tests/                    # self-tests of primitives
├── docs/plans/               # design records
├── package.json              # ESM, source-only exports
├── tsconfig.json
└── README.md
```

No `dist/`, no `prepublish`. Consumers are Next.js / Vite / Jest projects that already transpile TS from `node_modules`.

## Type design — generic over game-specific

The core never enumerates a game's question types. Instead:

```ts
export interface Question<TType extends string = string> {
  id: string;
  type: TType;
  concept: string;
  prompt: string;
  correctAnswer: string;
  distractors?: string[];
  formula?: string;
  interactive?: InteractiveData;
}

export interface InteractiveData {
  code?: string;
  [key: string]: unknown;
}
```

Consumers narrow:

```ts
type CSQuestionType = 'vocabulary' | 'trace_variables' | ...;
interface CSInteractive { variantData?: ...; outputData?: ...; }
type CSQuestion = Question<CSQuestionType> & { interactive?: CSInteractive };
```

Core owns **shapes and algorithms** (mastery math, PRNG, validation loops). Game-specific values flow in as config or type parameters.

## WF harness — pure core + framework adapters

**Pure validators** (`validators.ts`) take a `WFHarnessConfig` and return `ValidationResult[]` records. No describe/it/expect.

**Adapters** (`vitest.ts`, `jest.ts`) wrap each result group in that framework's `describe`/`it`/`expect` calls.

```ts
export interface WFHarnessConfig<TType extends string> {
  registeredTypes: readonly TType[];
  renderInteractiveCases: readonly TType[];
  interactivePayloadMap: Record<TType, PayloadSpec | null>;
  questionPool: Question<TType>[];
  conceptTree: ConceptNode[];
  generators: Generator[];
  quizClientPath?: string;
  renderPatternFor?: (type: string) => RegExp;
}
```

Consumer test file becomes ~30 lines of config.

## Risks

- **Vitest-in-a-package** can confuse IDE type resolution. Mitigation: `vitest` as `devDependency`, only imported inside `wf-harness/vitest.ts` adapter.
- **Group 2 (render-dispatch scanning)** reads a file path at test time. `quizClientPath` resolves relative to `process.cwd()`; documented in README.
- **Git-URL installs skip prepublish scripts.** Source-only avoids this entirely.
- **Jest vs Vitest assertion semantics** mostly compatible, but deep-equal on `expect().toEqual()` behaves identically, and that's all we need.
