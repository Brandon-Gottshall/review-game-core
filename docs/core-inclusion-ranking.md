*Last updated: 2026-04-19*

This table normalizes the ideas surfaced across the Codex threads into ranked candidates for `review-game-core` ownership.

Use `[/Volumes/EmmaSSD/Development/Personal/VSU/review-games/docs/codex-thread-intent-index.md](/Volumes/EmmaSSD/Development/Personal/VSU/review-games/docs/codex-thread-intent-index.md)` for thread-by-thread provenance. This document deduplicates overlapping threads and answers a narrower question:

> If an idea were promoted into shared core, how relevant would that be, and how useful would it actually be?

## Scoring

- `Relevance`: natural fit for shared core ownership.
- `Usefulness`: leverage across multiple games if centralized.
- Scale: `1` low, `5` high.
- `Recommendation` meanings:
  - `Core now`: core should actively own the runtime API, policy, or canonical doc now.
  - `Core docs/policy`: the shared doctrine belongs in core, even if the implementation seam stays thin.
  - `Boundary only`: core should document the seam and maybe expose helper types, but the concrete behavior stays above core.
  - `Keep local`: useful idea, but not a shared-core concern.
